Interrupt Handling Flowchart

Step-by-step decision flow when a hardware interrupt occurs

Legend

Terminal (Start/End)
Process / Action
Decision
ISR Execution
1
IRQ Assertion – Device signals the PIC/APIC via IRQ line when I/O completes or needs attention.
2
IF Flag Check – CPU checks the Interrupt Flag (IF) before accepting interrupts.
3
Context Save – PC, flags, and registers pushed to stack; kernel mode activated.
4
IVT Lookup – Interrupt Vector Table maps IRQ number → ISR address.
5
ISR Execution – Data is transferred, waiting process is woken via semaphore.
6
IRET – EOI sent to PIC, context restored, interrupted process resumes.

Polling vs Interrupt-Driven I/O

Comprehensive comparison of the two approaches with animated CPU utilization charts

Criterion 🔄 Polling ⚡ Interrupt-Driven
CPU Utilization High – CPU busy-waits Low – CPU free for other tasks
Latency Variable (depends on poll frequency) Near-instant (immediate response)
Complexity Simple implementation Moderate (ISR, IVT, context save)
Scalability Poor – degrades with more devices Excellent – handles many devices
Power Usage High – constant CPU activity Low – CPU can sleep between events
Best For High-frequency, predictable I/O Low-frequency, unpredictable I/O
Examples Game controllers, real-time ADC Disk, keyboard, network cards
Overhead Wasted cycles on status checks Context switch cost per interrupt
